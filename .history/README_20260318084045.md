# Internal Workflow Management Platform

A small production-oriented full-stack web application for managing internal tasks through a simple workflow.

## Overview

The platform supports:

- Task creation and management
- Task status transitions
- Task filtering by status
- Basic audit information (`created_at`, `updated_at`)

The application is implemented as two separated services:

- **Backend:** Django + Django REST Framework + SQLite
- **Frontend:** Vue 3 (CDN-based) + HTML/CSS/JavaScript, served by Nginx

## Architecture

### Backend

The backend is implemented with **Django** and **Django REST Framework**.

Main responsibilities:

- Expose RESTful APIs for task management
- Persist task data in SQLite
- Enforce workflow transition rules
- Provide filtering and basic search capabilities

#### Core model

The central domain model is `Task`, with the following fields:

- `id`
- `title`
- `description`
- `status`
- `created_at`
- `updated_at`

### Workflow rules

The workflow is intentionally simple and explicit:

- `OPEN -> IN_PROGRESS`
- `IN_PROGRESS -> COMPLETED`

Also allowed:

- `OPEN -> OPEN`
- `IN_PROGRESS -> IN_PROGRESS`
- `COMPLETED -> COMPLETED`

Not allowed:

- `OPEN -> COMPLETED`
- `IN_PROGRESS -> OPEN`
- `COMPLETED -> OPEN`
- `COMPLETED -> IN_PROGRESS`

Workflow validation is enforced on the backend, not only in the UI.

### Frontend

The frontend is implemented as a lightweight Vue-based interface using the Vue 3 CDN.

Views included:

- Task listing page
- Task creation form
- Status filtering
- Task detail page
- Task status update action

The frontend communicates with the backend using `fetch()` and JSON APIs.

### Containerization

The application is containerized using Docker and separated into two services:

- `backend`
- `frontend`

The frontend is served through **Nginx**, while the backend runs Django.

## **API Overview**

### Main endpoints

#### List tasks

`GET /api/tasks/`

#### Create task

`POST /api/tasks/`

Example payload:

```JSON
{
"title": "Implement API",
"description": "Create task endpoints",
"status": "OPEN"
}
```

#### Retrieve task details

`GET /api/tasks/<id>/`

#### Update task status

`PATCH /api/tasks/<id>/`

Example payload:

```JSON
{
"status": "IN_PROGRESS"
}
```

#### Filter tasks by status

`GET /api/tasks/?status=OPEN`

#### Supported statuses

- `OPEN`

- `IN_PROGRESS`

- `COMPLETED`

## How to Run the Application

### Run with Docker

#### From the project root:

```bash
docker compose up --build
```

Application URLs:

- Frontend: `http://127.0.0.1:8080`

- Backend API: `http://127.0.0.1:8000/api/tasks/`

- Django Admin: `http://127.0.0.1:8000/admin/`

To stop the containers:

```bash
docker compose down
```

## Admin Access

To create a Django admin user:

```bash cd backend
python manage.py createsuperuser
```

Then log in at:

`http://127.0.0.1:8000/admin/`

## Testing

Basic backend API tests are included.

To run tests:

```bash cd backend
python manage.py test
```

The tests cover:

- task listing

- task creation

- valid status transitions

- invalid status transitions


Trade-offs

This implementation intentionally favors clarity and maintainability over feature breadth.

Current trade-offs include:

SQLite instead of PostgreSQL

Vue CDN approach instead of a full SPA build pipeline

no authentication or role management

no pagination

no background jobs

no production-grade WSGI/ASGI server such as Gunicorn

These choices helped keep the solution small, understandable, and aligned with the assignment scope.

Potential Improvements

Given more time, the following would be the next improvements:

add authentication and authorization

add user ownership for tasks

move transition logic into a dedicated service layer

add pagination for task lists

add stronger form validation and nicer error handling in the UI

add full CI pipeline for tests and linting

replace SQLite with PostgreSQL

serve Django using Gunicorn in Docker

add environment-based configuration

add API schema documentation

improve frontend componentization with a bundler-based Vue setup

add end-to-end tests

Notes

This project was designed to be small but production-oriented, with emphasis on:

backend correctness

clear workflow validation

simple but usable frontend experience

clean project structure

straightforward local setup

readable documentation
